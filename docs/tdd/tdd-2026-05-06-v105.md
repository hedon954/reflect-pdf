## O15 — 字符级流式

### 问题剖析

v1.0.4 的 `translate_sentence_streaming` 用的是 `extract_complete_string_fields`，它只在字符串闭合后发射：

```rust
// v1.0.4 行为
// 1. SSE chunk 累计 raw="{\"translation\": \"你好"      → 不发射
// 2. SSE chunk 累计 raw="{\"translation\": \"你好世界" → 不发射
// 3. SSE chunk 累计 raw="{\"translation\": \"你好世界\"" → 终于发射！
```

闭合 `"` 是流的最后一帧，所以"流式"实际上等同于"非流式"。

### 设计

引入第二个提取器 `extract_streaming_string_value(buf, key) -> Option<String>`，专门用于**已知 key、希望逐字读取 value** 的场景：

```rust
/// Returns the in-progress value of `key` in `buf`, even if its closing quote
/// hasn't arrived yet. Returns `None` if the key hasn't been seen or its
/// value hasn't started. The returned string is fully decoded (escapes
/// resolved) up to the most recent UTF-8 safe position.
pub fn extract_streaming_string_value(buf: &str, key: &str) -> Option<String>;
```

实现要点：

1. 用 `format!("\"{key}\"")` 在 buffer 里**首次出现**位置定位 key（model 不会重复 emit 同名 key）
2. 跳过空白、`:`、空白、`"`，到达 value 的第一个字符位置
3. 复用一个 `read_json_string_partial` 工具：扫描到下一个未转义的 `"` 就**完整返回**；扫到末尾就**返回当前累积的字符串**（视为还在写）
4. 不允许把"闭合后又被截断"的字符串当作 partial（实际上 LLM 不会这样输出，但代码上保证健壮）

```rust
fn read_json_string_partial(buf: &str, start: usize) -> Option<String> {
    let mut out = String::new();
    let mut chars = buf[start..].char_indices();
    while let Some((_, c)) = chars.next() {
        match c {
            '"' => return Some(out),         // closing quote arrived
            '\\' => { /* same escape handling as read_json_string */ }
            other => out.push(other),
        }
    }
    Some(out)  // still streaming; return what we have so far
}
```

注意与 `read_json_string` 的区别：后者在闭合 `"` 缺失时返回 `None`（用于"完整字段"扫描，不能误报）；前者**始终返回**当前累积值（用于"已知字段、增量读取"）。

### 调用点

`translate_sentence_streaming` 重构（仅展示 closure 部分）：

```rust
let mut last_emitted_translation = String::new();
let raw_buf = self.stream_completion(&url, &body, |raw, _scratch| {
    // 字符级流式：每次都看最新累积值，差异即可发射
    let Some(current) = extract_streaming_string_value(raw, "translation") else {
        return;  // translation 字段还没开始
    };
    if current != last_emitted_translation {
        last_emitted_translation = current.clone();
        on_progress(TranslationResult {
            word: sentence.to_string(),
            context_sentence_translation: current,
            source: "llm".to_string(),
            ..Default::default()
        });
    }
}).await?;
```

每次 SSE chunk 到达时，extractor 都会重新读到 translation 字段当前的全部已收字符（自动包含上一次的内容 + 新增字符）。差异比较防止重复发射。

### emit 频率

LLM token 产生速率约 30–80 tok/s，每次 SSE delta 通常包含 1–3 token。1 token ≈ 1–3 中文字符或 1 英文字母。所以一次句子翻译期间 callback 大约会被调用 30–100 次——SwiftUI 完全够用，不会卡顿。

## O16 / O17 — Breakdown 数据模型

### Rust 数据结构

```rust
// domain/translation/entity.rs

#[derive(Debug, Clone, Default, uniffi::Record, serde::Serialize, serde::Deserialize)]
pub struct SentenceChunk {
    /// English sentence fragment.
    pub original: String,
    /// Translation of `original` into the target language.
    pub translation: String,
    /// Why this fragment is translated this way (word choice, context).
    pub explanation: String,
    /// Optional grammar analysis. Empty when the fragment has no
    /// noteworthy grammatical structure.
    #[serde(default)]
    pub grammar: String,
}

#[derive(Debug, Clone, Default, uniffi::Record, serde::Serialize, serde::Deserialize)]
pub struct TranslationResult {
    // ... existing fields
    /// Per-fragment breakdown for sentence translation. Empty for word mode
    /// or for short / simple sentences. Filled at end of stream.
    #[serde(default)]
    pub sentence_breakdown: Vec<SentenceChunk>,
}
```

`#[serde(default)]` 关键：旧缓存条目反序列化时拿到空 `Vec`，无 schema 迁移负担。

### Prompt 设计

```text
You are a professional translator and language tutor. Translate the following English text to {lang} AND, if it is a long or complex sentence, break it down to help the reader understand.

Text: "{sentence}"

Rules:
1. Provide a natural, fluent translation in `translation`.
2. Preserve meaning and tone of the original.
3. If OCR errors exist, fix them in your translation.
4. Decide whether to break the sentence down:
   - Short / simple sentences (≤10 words, no clauses) → leave `breakdown` as an empty array.
   - Long / complex sentences (multiple clauses, inversion, parallel structure, complicated adverbials, etc.) → split into 2–5 logical fragments.
5. For each fragment in `breakdown`:
   - `original`: the English fragment, exactly as in the source.
   - `translation`: that fragment's translation in {lang}.
   - `explanation`: in {lang}, briefly explain word choices / contextual meaning. Keep it under 2 sentences.
   - `grammar`: ONLY fill this when the fragment contains a grammatically noteworthy structure (subordinate clause, inversion, subjunctive mood, parallel structure, participle clause, nested complex structures, etc.). Leave it as an empty string for plain SVO fragments. Be concise — 1 to 3 sentences in {lang}, naming the structure and explaining its role.

Respond with ONLY valid JSON in this exact format:
{
  "translation": "<full translation in {lang}>",
  "breakdown": [
    {
      "original": "<English fragment>",
      "translation": "<{lang} translation of the fragment>",
      "explanation": "<{lang} explanation>",
      "grammar": "<{lang} grammar analysis OR empty string>"
    }
  ]
}
```

### 解析策略

由于 breakdown 是数组，且我们目前**不做数组级流式**（O16 以外的 Future Work 部分），实现简单：

1. 流式过程中，**只**用 `extract_streaming_string_value` 关注 `translation` 字段（持续 emit）
2. 流结束后，对累计的 `raw_buf` 做一次严格 `serde_json::from_str::<SentencePromptJson>(...)`，拿到 `translation` + `breakdown`
3. 把 breakdown 装进最终的 `TranslationResult`，由 use case / api 层 emit 一次"终态"

```rust
#[derive(Deserialize)]
struct SentencePromptJson {
    translation: Option<String>,
    #[serde(default)]
    breakdown: Vec<SentenceChunkJson>,
}

#[derive(Deserialize)]
struct SentenceChunkJson {
    original: Option<String>,
    translation: Option<String>,
    explanation: Option<String>,
    #[serde(default)]
    grammar: String,
}
```

`Option<String>` + `unwrap_or_default()` 容错处理：模型偶尔漏字段时给空串而不是 panic。

### `translate_sentence_streaming` 新签名

```rust
// 旧（v1.0.4）
pub async fn translate_sentence_streaming(
    &self,
    sentence: &str,
    on_progress: StreamProgress,
) -> Result<String, LumenError>;

// 新（v1.0.5）
pub async fn translate_sentence_streaming(
    &self,
    sentence: &str,
    on_progress: StreamProgress,
) -> Result<TranslationResult, LumenError>;
```

返回 `TranslationResult` 让 caller 拿到完整的 translation + breakdown，避免 api.rs 里再做一次手工拼接（v1.0.4 那段 `Ok(TranslationResult { ..., context_sentence_translation: translation, ... })` 可以删掉）。

### api.rs 简化

```rust
#[uniffi::export(async_runtime = "tokio")]
pub async fn translate_sentence_streaming(
    sentence: String,
    callback: Arc<dyn TranslationStreamCallback>,
) -> Result<TranslationResult, LumenError> {
    let config = llm_config()?;
    let llm = LlmTranslator::new(config);
    let on_progress: StreamProgress = {
        let cb = callback.clone();
        Box::new(move |partial| cb.on_progress(partial))
    };
    llm.translate_sentence_streaming(&sentence, on_progress).await
}
```

### `breakdownChunkView`

```swift
@ViewBuilder
private func breakdownChunkView(index: Int, chunk: SentenceChunk) -> some View {
    VStack(alignment: .leading, spacing: 6) {
        HStack(alignment: .firstTextBaseline, spacing: 6) {
            Text("\(index).")
                .font(.callout.bold())
                .foregroundStyle(.tertiary)
            VStack(alignment: .leading, spacing: 4) {
                Text(chunk.original)
                    .font(.system(.callout, design: .monospaced))
                    .foregroundStyle(.secondary)
                Text(chunk.translation)
                    .font(.callout)
                    .foregroundStyle(.primary)
            }
        }
        if !chunk.explanation.isEmpty {
            Text(chunk.explanation)
                .font(.caption)
                .foregroundStyle(.secondary)
                .padding(.leading, 16)
        }
        if !chunk.grammar.isEmpty {
            HStack(alignment: .top, spacing: 6) {
                Text("语法")
                    .font(.caption2.bold())
                    .padding(.horizontal, 6)
                    .padding(.vertical, 2)
                    .background(.blue.opacity(0.15), in: Capsule())
                    .foregroundStyle(.blue)
                Text(chunk.grammar)
                    .font(.caption)
                    .foregroundStyle(.primary)
            }
            .padding(.leading, 16)
        }
    }
}
```

### 流式期间的渲染

- **译文流入阶段**：`sentence_breakdown` 为空（流式中途不发射 breakdown），只显示「原文 + 译文（增长中）+ 正在生成…」
- **流式结束**：`sentence_breakdown` 在最终 emit 中填入，UI 自动追加拆解节
- 如果是短句，`breakdown` 为空，UI 与 v1.0.4 一致（无拆解节）

`hasAnyContent` 判定逻辑保持不变（已涵盖 `contextSentenceTranslation` 非空这条），无需修改。

## 兼容与回退

- **API**：`TranslationResult` 新增字段、未删除字段；`SentenceChunk` 通过 UniFFI Record 新增到 Swift。无破坏性变更。
- **缓存**：`#[serde(default)]` 处理旧条目；句子翻译当前不写缓存，所以即便有破坏也只影响单词缓存——单词缓存里没 breakdown 字段，反序列化得空 Vec，正常。
- **数据库 schema**：未变更。
- **回退**：如新 prompt 在某些模型上输出不稳定（例如 breakdown 字段不规范），最终的 `serde_json::from_str` 会失败，整个请求会冒泡为 `SerializationError`——上层 `api.rs::translate_sentence_streaming` 会冒泡到 Swift 层显示错误信息。监控这点；必要时可临时把 prompt 退回 v1.0.4 的简版（只要 translation 一个字段）。

## 后续改进（不在本期）

1. **chunk 级流式**：当 LLM 流式输出每个 breakdown 项时即时弹入。需要在 streaming.rs 加一个 `extract_completed_array_objects(buf, key)`，按 `{...}` 边界返回完整对象数组。预计 v1.0.6 视用户反馈再决定。
2. **持久化 breakdown 到笔记**：保存句子翻译时把 breakdown 序列化追加到笔记 `note` 字段；若变 schema 则按 v1.0.3 的 sqlx migrate 流程进行。
3. **配置化的拆解阈值**：当前由 LLM 自行判断；未来可引入"是否启用拆解"的开关，让用户在设置里关掉以省 token。
