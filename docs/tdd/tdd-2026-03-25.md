## 1. 翻译失败原因展示

### 1.1 问题根因

LLM 翻译失败的路径有两种：

| 情形                                  | 表现                                                | 原因          |
| ------------------------------------- | --------------------------------------------------- | ------------- |
| **完全失败**（LLM + MyMemory 均失败） | `translate()` 抛出 `LumenError`，Swift `catch` 捕获 | 两级均 `Err`  |
| **LLM 失败 + MyMemory 兜底成功**      | `translate()` 返回 `Ok`，Swift `catch` **不执行**   | Rust 静默降级 |

截图中「基础翻译」即第二种情形——LLM 调用失败但 MyMemory 兜底返回了结果，Swift 侧认为翻译成功，没有渠道展示 LLM 失败原因。

```rust
impl LumenError {
    pub fn user_hint_zh(&self) -> String {
        match self {
            LumenError::ConfigNotInitialized =>
                "LLM 未就绪：请先在「设置」中填写 API Base URL、API Key 与模型并保存。",
            LumenError::LlmApiError { message } => format!("LLM 接口失败：{}", message),
            // ... 其他变体
        }
    }
}
```

**`lumen-pdf-core/src/domain/translation/service.rs`**

```rust
// Level 2: LLM
let llm_failure_note: Option<String> = match self.llm.translate(...).await {
    Ok(mut result) => {
        result.llm_error_message = String::new();  // 显式清空
        let _ = self.cache.set(...);
        return Ok(result);
    }
    Err(e) => Some(e.user_hint_zh()),  // 记录失败原因
};

// Level 3: fallback
let mut result = self.fallback.translate(...).await?;
if let Some(msg) = llm_failure_note {
    result.llm_error_message = msg;   // 降级成功，但携带 LLM 错误信息
}
Ok(result)
```

## 2. 高亮 / 划线 Cmd+Z 撤销

### 2.1 设计要点

- **作用范围**：仅对「自由标注」（`__fh` 高亮 / `__fu` 划线）生效；单词本高亮（按 `entryId` 标记）不纳入撤销，避免与单词本数据库状态不一致。
- **Undo Manager**：直接使用 `PDFView.undoManager`（AppKit/PDFKit 原生），与系统菜单「编辑 > 撤销 xxx」保持一致。
- **成对注册**：每次 `addFreeAnnotation` 操作后，注册 Undo 闭包；Undo 执行时内部注册 Redo 闭包，保证多级撤销/重做可用。

### 2.2 `FreeAnnotationSnapshot`

```swift
private struct FreeAnnotationSnapshot {
    let bounds: CGRect
    let color: NSColor
    let tag: String          // "__fu" | "__fh"
    init(ann: PDFAnnotation) { ... }
    var subtype: PDFAnnotationSubtype {
        tag == "__fu" ? .underline : .highlight
    }
}
```

`PDFAnnotation` 在 Swift 中没有可靠的 `type` 属性（返回 `String?`），改由 `tag`（`userName`）推断类型。

### 2.3 核心注册逻辑

```swift
// addFreeAnnotation 末尾
var added: [PDFAnnotation] = []
var removedSnapshots: [FreeAnnotationSnapshot] = []
// ... 执行 add / remove ...
if !added.isEmpty || !removedSnapshots.isEmpty {
    registerUndoAnnotationMutation(page:added:removedSnapshots:label:)
}

// Undo 闭包
undo.registerUndo(withTarget: self) { [weak self] _ in
    // 1. 移除 added 中每个标注
    // 2. 从 removedSnapshots 重建原标注 → restored
    // 3. 注册 Redo（即再次执行 add 操作）
}
undo.setActionName(label)  // 系统菜单显示「撤销 高亮」/「撤销 划线」
```

`makeAnnotation(bounds:type:color:tag:page:)` 改为返回 `PDFAnnotation`（原为 `Void`），以便 Undo/Redo 持有引用。

### 2.4 Cmd+Z 转发

SwiftUI 不直接管理 `undoManager`，在 `PDFReaderView` 的 `ZStack` 内添加隐藏按钮：

```swift
Button("") {
    NSApp.sendAction(Selector(("undo:")), to: nil, from: nil)
}
.keyboardShortcut("z", modifiers: .command)
.frame(width: 0, height: 0).opacity(0)
```

`sendAction` 沿 AppKit 响应链向上传递，最终到达 `PDFView.undoManager`。
