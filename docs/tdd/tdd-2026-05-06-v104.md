## 总体架构

流式翻译的端到端调用链如下：

```
SwiftUI                       UniFFI                 Rust async
─────────────────             ──────────             ─────────────────
TranslationBubble                                    LlmTranslator
    │                                                    │
    │ onPartial closure                                  │ stream: true
    ▼                                                    ▼
PDFReaderView           ┌─────────────────────┐     reqwest.bytes_stream()
    │                   │ Foreign Callback     │          │
    │                   │ (with_foreign trait) │          │
    │   awaits result   │                      │          │ chunks
    └──────────────────►│ on_progress(...)     │◄─────────┤
                        │   ↓                  │          │
                        │ MainActor.run        │          │ SseAccumulator
                        └─────────────────────┘           │ + extract_complete_string_fields
                                                          │
                                                          ▼
                                                     TranslationDomainService
                                                     (cache → llm streaming → fallback)
```

关键约束：
- **Rust async 在 tokio 工作线程上消费 SSE**，每个分块通过 UniFFI 回调推送给 Swift
- **Swift 一律在 `MainActor` 上更新 UI**（`TranslationStreamReceiver` 自动 hop）
- **域层（`domain/`）零 I/O**：仅扩展 `Translator` trait，新增的 `translate_streaming` 默认方法靠 `repository.rs` 提供

## O14 — HTTP 客户端共享与超时

### 设计

```rust
// infrastructure/translator/http_client.rs
static HTTP_CLIENT: OnceLock<Client> = OnceLock::new();

pub fn shared_client() -> &'static Client {
    HTTP_CLIENT.get_or_init(|| {
        Client::builder()
            .connect_timeout(Duration::from_secs(3))
            .timeout(Duration::from_secs(30))
            .pool_idle_timeout(Duration::from_secs(60))
            .build()
            .unwrap_or_else(|_| Client::new())
    })
}
```

### 关键决策

| 决策 | 取值 | 理由 |
| --- | --- | --- |
| `connect_timeout` | 3s | 国内网关 RTT 200–800ms，3s 足够区分"慢"和"挂"，超过即触发兜底 |
| `timeout` | 30s | 覆盖最长一次词级翻译流式生命周期（≤ 270 tokens × 30tok/s ≈ 9s + buffer） |
| `pool_idle_timeout` | 60s | 长时间不用的连接释放，避免 keep-alive 已被服务端断开却仍尝试复用 |
| 失败兜底 | `unwrap_or_else(\|_\| Client::new())` | 系统 TLS 出错时不让进程崩溃，让翻译以"无超时"姿态继续工作（极端情况） |

LLM / fallback 翻译器都改为引用同一个 `shared_client()`，连接池 + TLS 会话复用从此跨翻译器共享。

## O12 — 流式翻译核心：SSE 解析 + 增量 JSON

### SSE 行重组（`SseAccumulator`）

`reqwest::bytes_stream()` 给的是字节块，**不保证按行对齐**。一次 chunk 可能：
- 包含多个完整行 + 一行尾巴
- 把一行劈成两半（甚至 UTF-8 多字节字符也劈开）

`SseAccumulator` 维护一个未消费缓冲区：

```rust
pub struct SseAccumulator {
    pending: String,
}

impl SseAccumulator {
    pub fn feed(&mut self, chunk: &str) -> SseChunkOutcome {
        self.pending.push_str(chunk);
        let mut content = String::new();
        let mut done = false;
        while let Some(idx) = self.pending.find('\n') {
            let line = self.pending[..idx].trim_end_matches('\r').to_string();
            self.pending.drain(..=idx);
            if let Some(payload) = line.strip_prefix("data:") {
                let payload = payload.trim();
                if payload == "[DONE]" { done = true; continue; }
                if payload.is_empty() { continue; }
                if let Ok(parsed) = serde_json::from_str::<SseChatChunk>(payload) {
                    if let Some(delta) = parsed.choices.into_iter().next().and_then(|c| c.delta.content) {
                        content.push_str(&delta);
                    }
                }
                // 静默忽略 keep-alive 注释、event 行、vendor 自定义事件
            }
        }
        SseChunkOutcome { content_deltas: content, done }
    }
}
```

UTF-8 多字节切边问题在 LlmTranslator 调用层处理：

```rust
let mut byte_buf: Vec<u8> = Vec::new();
while let Some(item) = stream.next().await {
    let bytes = item?;
    byte_buf.extend_from_slice(&bytes);
    let valid_len = match std::str::from_utf8(&byte_buf) {
        Ok(s) => s.len(),
        Err(e) => e.valid_up_to(),
    };
    if valid_len == 0 { continue; }
    let valid_str = unsafe { std::str::from_utf8_unchecked(&byte_buf[..valid_len]) }.to_string();
    byte_buf.drain(..valid_len);
    let outcome = sse.feed(&valid_str);
    // ...
}
```

`valid_up_to()` 截到最后一个完整 UTF-8 字符，剩下的不完整字节留在缓冲区等下一次。

### 增量 JSON 字段提取（`extract_complete_string_fields`）

LLM 流式返回的是**还没写完的 JSON**，例如：

```json
{"word": "investigate", "phonetic": "/ɪnˌv
```

我们需要从这种"半成品"里安全抽出**已闭合**的 `"key": "value"` 对（这里只能拿到 `word`，`phonetic` 还没收尾）。

实现要点：

1. **不依赖 `serde_json`**——它只接受完整 JSON，半成品会全部失败
2. **字节扫描 + UTF-8 安全字符迭代**：JSON 结构字符（`"` `:` `\\` 等）都是 ASCII < 0x80，永远不会出现在 UTF-8 多字节序列的延续字节里，所以可以按字节扫描定位
3. **完整转义支持**：`\"` `\\` `\n` `\r` `\t` `\b` `\f` `\/` `\uXXXX` 都正确解析
4. **遇到第一个未完成的值就停**，不会把"还没收尾的字段"误当作已完成

```rust
pub fn extract_complete_string_fields(buf: &str) -> Vec<(String, String)> {
    let bytes = buf.as_bytes();
    let mut out = Vec::new();
    let mut i = 0;
    while i < bytes.len() {
        if bytes[i] != b'"' { i += 1; continue; }
        let Some((key, after_key)) = read_json_string(buf, i + 1) else { break };
        i = after_key;
        i = skip_ws(bytes, i);
        if i >= bytes.len() || bytes[i] != b':' { continue; }
        i = skip_ws(bytes, i + 1);
        if i >= bytes.len() || bytes[i] != b'"' { continue; }
        let Some((value, after_val)) = read_json_string(buf, i + 1) else { break };
        out.push((key, value));
        i = after_val;
    }
    out
}
```

`read_json_string` 用 `char_indices()` 迭代——保证 IPA 音标、CJK、emoji 等多字节字符**逐字符**正确累加，不会被字节级 `as char` 错误切割。

## O12 — 域层流式编排

### `Translator` trait 默认方法

```rust
// domain/translation/repository.rs
pub type StreamProgress = Box<dyn FnMut(TranslationResult) + Send>;

#[async_trait::async_trait]
pub trait Translator: Send + Sync {
    async fn translate(&self, word: &str, sentence: &str)
        -> Result<TranslationResult, LumenError>;

    /// 默认实现：调用 `translate` 拿全量结果后单次 emit。
    /// 这样 fallback 翻译器不需要自己实现流式即可融入流式管线。
    async fn translate_streaming(
        &self,
        word: &str,
        sentence: &str,
        mut on_progress: StreamProgress,
    ) -> Result<TranslationResult, LumenError> {
        let result = self.translate(word, sentence).await?;
        on_progress(result.clone());
        Ok(result)
    }
}
```

`LlmTranslator` 在自己的 `impl Translator` 里**覆盖**这个方法，实现真正的 SSE 消费 + 增量发射。

### `TranslationDomainService::translate_streaming`

```rust
pub async fn translate_streaming(
    &self,
    request: TranslationRequest,
    on_progress: StreamProgress,
) -> Result<TranslationResult, LumenError> {
    let shared = Arc::new(Mutex::new(on_progress));
    let emit = |r: TranslationResult| {
        if let Ok(mut cb) = shared.lock() { (cb)(r); }
    };

    // 1) 缓存命中：单次 emit + 返回
    if let Some(mut cached) = self.cache.get(&word_lower, &hash)? {
        cached.source = "cache".into();
        emit(cached.clone());
        return Ok(cached);
    }

    // 2) LLM 流式：包一层 forwarder 改写 source
    let forwarder: StreamProgress = {
        let s = shared.clone();
        Box::new(move |mut p| {
            p.source = "llm".into();
            if let Ok(mut cb) = s.lock() { (cb)(p); }
        })
    };
    match self.llm.translate_streaming(&request.word, &request.sentence, forwarder).await {
        Ok(mut result) => {
            result.source = "llm".into();
            let _ = self.cache.set(&word_lower, &hash, &result);
            emit(result.clone());  // 终态 emit
            return Ok(result);
        }
        Err(e) => llm_failure_note = Some(e.user_hint_zh()),
    }

    // 3) 兜底：单次 emit
    match self.fallback.translate(...).await {
        Ok(mut r) => { r.source = "fallback".into(); emit(r.clone()); Ok(r) }
        Err(e) => { /* 失败状态结构体，emit + 返回 */ }
    }
}
```

**`Arc<Mutex<StreamProgress>>` 的作用**：让我们既能把 `on_progress` 借给 LLM 流式（通过 forwarder），又能在流式完成后**继续调用它**做终态 emit。`Mutex` 不存在并发风险（emission 严格串行），用纯 std 库即可，避免引入新依赖。

## O12 — UniFFI 跨语言流式

### Rust 侧：`#[uniffi::export(with_foreign)]` 回调 trait

```rust
#[uniffi::export(with_foreign)]
pub trait TranslationStreamCallback: Send + Sync {
    fn on_progress(&self, partial: TranslationResult);
}

#[uniffi::export(async_runtime = "tokio")]
pub async fn translate_streaming(
    request: TranslationRequest,
    callback: Arc<dyn TranslationStreamCallback>,
) -> Result<TranslationResult, LumenError> {
    // 构建 use case + 把 callback 适配成 StreamProgress
    let on_progress: StreamProgress = {
        let cb = callback.clone();
        Box::new(move |partial| cb.on_progress(partial))
    };
    TranslationUseCase::new(...)
        .translate_streaming(request, on_progress).await
}
```

### Swift 侧：自动生成的协议

UniFFI 生成的 `lumen_pdf_core.swift` 中：

```swift
public protocol TranslationStreamCallback : AnyObject {
    func onProgress(partial: TranslationResult)
}

public func translateStreaming(
    request: TranslationRequest,
    callback: TranslationStreamCallback
) async throws -> TranslationResult
```

### Swift 侧适配层：`TranslationStreamReceiver`

```swift
private final class TranslationStreamReceiver: TranslationStreamCallback {
    private let onPartial: @MainActor (TranslationResult) -> Void

    init(onPartial: @escaping @MainActor (TranslationResult) -> Void) {
        self.onPartial = onPartial
    }

    func onProgress(partial: TranslationResult) {
        let snapshot = partial
        Task { @MainActor in
            self.onPartial(snapshot)
        }
    }
}
```

**线程模型**：Rust 在 tokio worker 上调用 `cb.on_progress(partial)`，UniFFI 在 Swift 侧执行 `receiver.onProgress(partial:)`——这次调用**可能不在 MainActor 上**。`Task { @MainActor in ... }` 把 UI 更新强制 hop 到主线程，符合 SwiftUI 与 `@Published` 的约束（参见 `swift-conventions.mdc`）。

## O13 — 气泡渐进渲染

### 状态机变化

| 旧逻辑 | 新逻辑 |
| --- | --- |
| `if isLoading` → 大 spinner | `if result has any content` → 内容视图（含 streamingBanner） |
| `else if let result` → 内容视图 | `else if isLoading` → 大 spinner（首字到来前） |
| `else` → 错误/空 | `else` → 错误/空（不变） |

`hasAnyContent(_:)` 仅检查"非 word 字段"是否有内容（`word` 字段恒非空——它默认为用户选中的原文，不是有效的"加载完成"信号）。

### 流式期间不显示保存按钮

```swift
if !isLoading {
    Divider()
    footer(result: result)
}
```

理由：用户在流式途中点击"保存"会把字段不全的部分结果写入数据库，污染单词本/笔记。等流式结束保证最终值再允许保存。

## O13 — 防竞态：requestId

`PDFReaderView.requestTranslation` 启动 Task 前捕获 `requestId = translationRequest?.id`。所有更新 `translationRequest` 的位置（`applyPartial`、最终成功、错误处理）都验证：

```swift
guard var req = translationRequest, req.id == requestId else { return }
```

场景覆盖：
- 用户关闭气泡 → `translationRequest = nil`，guard 失败，旧 callback 静默丢弃
- 用户选了新词 → `translationRequest.id` 变了，旧 callback guard 失败
- 流式完成的最终 emit 到达时 → 同上，避免覆盖新选词的结果

## 已知边界 / 后续改进

- **预热连接**（PRD 未列入）：进程冷启后第一次翻译仍要走 TCP+TLS 握手 200–800ms，可在 `initialize()` 时发一个轻量 HEAD 请求预热。预计未来版本视体感再决定。
- **缓存读 SQLite 阻塞 tokio worker**：当前 `cache.get(...)` 在 async 上下文中同步执行 rusqlite。单条查询 1–5ms，目前不构成瓶颈，但在并发高时可改为 `spawn_blocking`。
- **同请求 in-flight 去重**：用户双击同词会发两次请求。可在 use case 层加 `Mutex<HashMap<(word,hash), Shared<Future>>>` 去重，本期未做。
- **首字段策略**：当前实现"按字段完成度"分批 emit；未来可以改成"按 token 增量"流式更新某个长字段（如 `context_explanation` 一边生成一边出现），需要 UI 配合。
