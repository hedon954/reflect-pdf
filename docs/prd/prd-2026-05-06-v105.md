# LumenPDF — 产品需求文档 (PRD)

**版本**: v1.0.5 · **日期**: 2026-05-06

---

## 本次迭代概述 (v1.0.5)

v1.0.4 把翻译切到了流式输出，但**句子翻译**模式存在两个问题：

1. **流式没有真正生效**：实现上"等到字符串完整闭合才推送一次"，对长句而言等同于非流式（用户仍要等几秒空白）。
2. **缺少教学价值**：句子模式只给一段译文，没有告诉用户"为什么这么翻"——对长难句，用户希望理解断句、词义选择和复杂语法。

v1.0.5 围绕**句子翻译体验**做三项优化：

1. **真·流式输出**：句子译文按字符流式渲染，用户能看到译文一字一字写出来。
2. **句子拆解分析**：长难句自动拆成 2–5 个语义片段，每段配中文解释；短句不拆。
3. **语法分析**：拆解中遇到复杂语法（从句、倒装、虚拟语气、并列结构等）时附加语法说明；普通片段留空。

本版本不引入新业务对象，纯属对现有"句子翻译"流程的体验升级。

## 新增 / 变更功能需求

### O15 — 句子翻译真·流式输出

**问题**：

v1.0.4 的句子流式实现里，`extract_complete_string_fields` 只在 `"translation"` 字段的**闭合 `"`** 到达后才发射一次。LLM 在生成长句时，闭合引号是流的最后才出现的，所以用户体验上：

- 看不到"边写边出"
- 整段译文一次性闪现
- 与非流式无差异

**需求**：

- 句子模式下，**译文按字符流式可见**：用户每隔约 30–80ms 看到新字符出现
- 句子模式不再依赖"字段闭合"才推送，而是按 SSE delta 累积出的"当前 in-progress 值"持续 emit
- 单词模式继续保持"按字段完成"发射（短字段无需字符级流，避免闪烁）
- 缓存命中、兜底、失败路径行为不变

**预期效果**：

| 指标 | v1.0.4 | v1.0.5 |
| --- | --- | --- |
| 句子译文首字 | 翻译完成后整段闪现（≈ 2–5s） | 0.5–2s（≈ TTFT） |
| 句子译文逐字流式 | ❌（一次性显示） | ✅（按字符增量） |
| 单词模式 | 按字段（保持不变） | 按字段（保持不变） |

---

### O16 — 长难句拆解分析

**问题**：

句子翻译只给最终译文，对长难句来说不够"教学"——用户难以反向理解：

- 这个长句子是怎么断的？
- 这个词在这里为什么翻成这个意思？
- 哪里是主句、哪里是从句？

**需求**：

- LLM 输出格式扩展：句子模式响应里追加 `breakdown` 数组，每项包含：
  - `original`：英文原句的语义片段
  - `translation`：该片段对应的中文译文
  - `explanation`：用中文解释这段为什么这么翻（词义、语境、上下文选择）
  - `grammar`：可选，仅在该片段含值得讲的语法点时填写（详见 O17）
- LLM **自行判断**是否需要拆解：
  - 短句 / 简单句（如 ≤10 词、无从句）→ `breakdown = []`
  - 长句 / 含从句 / 含倒装 / 含并列 → 拆成 2–5 段
- 单词模式不需要 breakdown，prompt 不要求该字段
- breakdown 仅在气泡 UI 中展示；本期**不持久化到笔记**（保存按钮仍只存原文 + 译文）

**UI 渲染（句子模式）**：

```
┌─────────────────────────────────────────────────┐
│  原文：Although the rain was heavy, ...         │
│                                                  │
│  译文：尽管雨下得很大，我们还是按时到达了。     │
│                                                  │
│  ─── 拆解分析 ───                               │
│                                                  │
│  ① Although the rain was heavy,                 │
│     尽管雨下得很大，                             │
│     "Although" 是让步连词，引出让步状语从句。   │
│     [语法] 让步状语从句 + 主句的复合结构。      │
│                                                  │
│  ② we still arrived on time.                    │
│     我们还是按时到达了。                         │
│     "still" 与"although"呼应，强调对比。         │
└─────────────────────────────────────────────────┘
```

- 每个片段：原文（小号灰色等宽）+ 译文（主色）+ 解释（次级）
- 仅当 `grammar` 非空时显示「语法」小标签 + 内容
- 短句（`breakdown` 为空）时不显示拆解节，气泡保持 v1.0.4 简洁版式

### O17 — 复杂语法节

**问题**：

部分长难句的难点不在"词义"而在"语法结构"——例如倒装、强调句、虚拟语气、嵌套从句。仅靠词义解释覆盖不到这些情况。

**需求**：

- breakdown 中**每个片段独立判断**是否含复杂语法
- 复杂语法定义（提示给 LLM）：
  - 从句（定语从句、状语从句、名词性从句）
  - 倒装语序、强调句（`It is ... that`）
  - 虚拟语气（`were`、`would have`）
  - 并列结构（`not only ... but also`、`either ... or`）
  - 分词短语作状语 / 独立主格
  - 长复合结构（嵌套两层及以上）
- 简单片段（一般主谓宾、单一介词短语等）`grammar` 留空，UI 不显示
- 语法说明用中文，且**简洁**：1–3 句话点出结构名称 + 在该句中的作用
- LLM 自行判断粒度，不强求每段都有 grammar

**反例**（不应填 grammar 的情况）：

- "He went to school." → 无需语法分析
- "The book on the table" → 简单介词短语，无需

## 兼容性与回归

- **API 兼容**：`TranslationResult` 新增字段 `sentence_breakdown: Vec<SentenceChunk>`，使用 `#[serde(default)]`，**旧缓存无需迁移**（反序列化得到空数组）
- **数据库**：本次迭代**不涉及 schema 变更**
- **缓存策略**：句子翻译目前不进缓存（与 v1.0.4 一致），breakdown 也不入库
- **UniFFI 桥接**：新增 `SentenceChunk` Record；现有调用代码无需修改即可继续工作（默认值就是空数组）

## 验收标准

| 用例 | 期望行为 |
| --- | --- |
| 选中长句子（如 30 词）翻译 | 译文按字符流式出现；流结束后底部出现 2–5 段拆解，含解释 |
| 选中短句（≤10 词）翻译 | 译文流式显示，但无拆解节（breakdown 为空） |
| 长句中含明显从句 / 倒装 | 至少一个片段带有语法节 |
| 平铺直叙的长句 | 可能有拆解但语法节为空 / 极少 |
| 网络断开 | 30s 后兜底到 MyMemory（无 breakdown，与 v1.0.4 一致） |
| 流式中途切到新句 | 旧请求的进度被丢弃（requestId 防竞态，沿用 v1.0.4） |

## 不在本期范围内

- 拆解结果保存到笔记（用户可能希望，但本期先观察气泡内体验，避免提前做 schema 变更）
- 单词模式的拆解（v1.0.4 已有 `context_explanation`，重复价值低）
- 拆解项级别的流式（每个 chunk 出现时单独弹入）。本期 chunk 在流式结束后整体出现；若用户反馈"等待感强"，再做 v1.0.6 优化
